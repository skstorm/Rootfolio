# Anime Title Academy - 일반 기획서 보강 문서

본 문서는 기존 일반 기획서(anime_title_academy_concept.md)에서 누락되거나 보강이 필요한 항목을 정리합니다.

---

## 1. 누락된 UX/비즈니스 요소

### 1.1 온보딩 플로우
기존 기획서에 첫 실행 경험이 정의되지 않았습니다.
- 첫 실행 시 3단계 튜토리얼: "사진 선택 → 프리셋 선택 → 스크래치 체험"
- 샘플 이미지로 즉시 체험 가능하도록 구성 (사진 권한 요청 전 진입장벽 제거)
- 개인정보 수집 동의 및 이용약관 동의 화면 필수 포함

### 1.2 결과물 저장 및 갤러리
"생성 → 스크래치 → 끝"으로 플로우가 종료되며, 저장/재확인 기능이 없습니다.
- 로컬 갤러리에 생성 히스토리 저장
- 결과물 디바이스 저장 (카메라롤 내보내기)
- 히스토리에서 재생성(같은 사진 + 다른 프리셋) 기능

### 1.3 공유 기능
바이럴 성장의 핵심 요소인 공유 기능이 기획서에 없습니다.
- SNS 공유 (카카오톡, 인스타 스토리, 트위터 등)
- 공유 시 앱 다운로드 링크 자동 포함
- 공유 이미지에 워터마크 + 앱 로고 삽입으로 자연스러운 바이럴

### 1.4 에러/예외 상황 UX
- AI 서버 응답 실패 시 재시도 UI
- 네트워크 미연결 시 오프라인 안내
- AI가 부적절한 자막을 생성했을 때 "다시 생성" 버튼
- AI가 빈 문자열/의미 없는 텍스트 반환 시 폴백 처리

### 1.5 사용자 설정
- 자막 폰트 크기/위치 미세 조정
- 기본 언어 설정 (한국어/일본어/영어)
- 알림 설정
- 캐시 삭제

## 2. 보안 및 개인정보 (Critical)

### 2.1 개인정보 처리 방침
- 사용자 사진이 AI 서버로 전송됨 → 수집 동의, 보관 기간, 삭제 정책 필수
- 앱스토어/플레이스토어 제출 시 Privacy Policy URL 없으면 거절
- GDPR, 한국 개인정보보호법 준수

### 2.2 API 키 보호
- 클라이언트에 AI API 키 직접 포함 금지 (디컴파일 추출 가능)
- 프록시 서버를 통한 API 호출 구조 필요
- **백엔드 아키텍처가 기획서에 전혀 없음** → 결정 필요

### 2.3 이미지 전송 보안
- TLS 필수
- 서버 측 이미지 보관 여부/기간 정의
- 처리 후 즉시 삭제 여부 명시

### 2.4 미성년자 보호
- 애니메이션 테마 → 미성년자 사용 가능성 높음
- COPPA/아동 관련 규정 준수 여부 검토
- 연령 확인 절차 필요 여부 결정

### 2.5 얼굴 인식 데이터
- 사진에서 얼굴 처리 시 iOS는 "민감 데이터"로 분류
- App Store 심사 시 해당 고지 필요

## 3. 엣지 케이스

| 상황 | 현재 대응 | 필요 대응 |
|------|----------|----------|
| 초고해상도 이미지 (108MP) | 미정의 | 업로드 전 리사이즈 임계값 설정 (예: 2048px) |
| AI 빈 문자열 반환 | 미정의 | 재생성 옵션 + 폴백 자막 |
| 동일 사진 반복 생성 | 미정의 | 매번 다른 결과 생성 + 캐시 정책 결정 |
| 스크래치 도중 앱 종료 | 미정의 | 재진입 시 결과 즉시 표시 또는 처음부터 |
| 다국어 자막 길이 차이 | 미정의 | 폰트 자동 크기 조절, 줄바꿈 로직 |
| 광고 로딩 실패 | 미정의 | 광고 스킵 후 진행, 리워드는 대체 수단 제공 |
| 앱 업데이트 후 로컬 데이터 | 미정의 | DB 마이그레이션 전략 수립 |

## 4. 검증되지 않은 핵심 가정

### 가정 1: "AI가 유머 자막을 잘 생성할 것"
- **위험도**: Critical (앱의 핵심 가치)
- **검증 방법**: 사진 50장에 대해 실제 AI 자막 생성 → 사용자 10명 "웃기다" 판정 비율 측정
- **시점**: Phase 1 시작 전 PoC 필수

### 가정 2: "스크래치 UX가 바이럴을 유발할 것"
- **위험도**: High
- **검증 방법**: 프로토타입으로 스크래치 유/무 A/B 테스트 후 공유율 비교

### 가정 3: "무료 3종으로 충분한 체험 제공"
- **위험도**: Medium
- **검증 방법**: 프리셋 수별 전환율 시뮬레이션

### 가정 4: "Flutter에서 스크래치 60fps 가능"
- **위험도**: Medium
- **검증 방법**: Phase 2 시작 전 기술 스파이크(PoC) 수행

## 5. 미결정 사항 (Open Questions)

| 항목 | 영향 범위 | 결정 시점 |
|------|----------|----------|
| AI 자막 생성 모델 (Gemini/GPT/Claude) | API 비용, 응답 시간, 품질 | Phase 1 시작 전 |
| 백엔드 서버 필요 여부 (서버 vs 서버리스) | API 키 보호, 영수증 검증, 사용량 제한 | Phase 1 시작 전 |
| 타겟 플랫폼 (iOS+Android 동시 vs 우선) | 개발 범위, 테스트 매트릭스 | Phase 1 시작 전 |
| 사용자 인증 시점 (익명 → 결제 시 가입?) | 갤러리 동기화, 결제 연동 | Phase 1 설계 시 |
| 사진 보관/삭제 정책 | 개인정보 처리방침, 앱스토어 심사 | Phase 1 시작 전 |
| 월간 AI API 비용 예산 상한 | 무료 사용자 생성 횟수 제한 정책 | Phase 1 시작 전 |
| 무료 사용자 일일/월간 생성 횟수 제한 | 비용 통제, 유료 전환 유도 | Phase 1 설계 시 |

## 6. 추가 모듈화 제안

기존 4대 모듈 외에 다음 6개 모듈을 추가 분리할 것을 권장합니다.

| 모듈 | 필요성 | Phase |
|------|--------|-------|
| **Watermark Module** | 워터마크 삽입/제거가 Billing과 연동되므로 독립 모듈로 분리 필요 | Phase 1 |
| **ShareKit Module** | SNS 공유는 바이럴 핵심이며, 플랫폼별 분기 로직이 복잡 | Phase 1 |
| **Gallery Module** | 히스토리 저장/조회, 로컬 DB 관리 독립 | Phase 1 |
| **Billing Module** | AdManager와 인앱결제는 관심사가 다름, 영수증 검증 등 별도 관리 | Phase 4 |
| **Analytics Module** | 사용자 행동 추적, A/B 테스트, 수익 분석을 위한 이벤트 수집 | Phase 1 |
| **Onboarding Module** | 첫 실행 경험, 튜토리얼, 권한 요청 플로우 | Phase 1 |

## 7. 우선순위별 권장 액션

### Critical (Phase 1 시작 전 반드시)
1. 백엔드 아키텍처 결정 (API 키 보호, 결제 검증, 사용량 제한)
2. AI 서비스 선정 및 유머 자막 생성 PoC
3. 개인정보 처리 방침 및 콘텐츠 모더레이션 정책 수립

### High (Phase 1 설계에 포함)
4. ShareKit/Gallery 모듈을 Phase 1에 추가 (저장+공유 없으면 MVP가 아닌 기술 데모)
5. 에러 핸들링 전략을 모듈 인터페이스 수준에서 정의
6. 각 Phase의 Definition of Done 확정

### Medium (Phase 2 이전)
7. 상태관리 라이브러리 확정 (Riverpod 권장)
8. 무료 사용자 생성 횟수 제한 정책 결정
9. Flutter 스크래치 60fps 성능 PoC

### Scope 관리
10. Phase 3 범위를 "스타일 1종, 해상도 1종"으로 고정
11. 소셜/커뮤니티 기능은 MVP에서 명시적 제외 선언
12. 프리셋 추가 파이프라인 및 최대 수량 상한 정의
