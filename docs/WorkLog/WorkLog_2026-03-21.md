## 📌 완료 작업
### 1. LLM 모델 구조 점검 및 모델 선택 UX 추가
- 현재 앱이 `.env` 미설정 시 `gemini-2.5-flash-lite`를 기본 LLM으로 사용하도록 정리.
- 결과 화면에 드롭다운형 모델 선택 버튼을 추가하여 `빠름 / 밸런스 / 고품질` 3단계 모델 전환이 가능하도록 구조를 확장.
- 각 모델은 각각 `gemini-2.5-flash-lite`, `gemini-2.5-flash`, `gemini-2.5-pro`로 연결.
- `AiClient -> GeminiLlmDatasource -> TitleRepository -> TitleNotifier -> ResultPage` 흐름으로 선택 모델이 전달되도록 파이프라인을 수정.

### 2. 스크래치 꽃가루 이펙트 트리거 안정화
- 꽃가루 이펙트가 간헐적으로 재생되지 않는 현상을 분석한 결과, 기존 구현이 `isCleared` 불리언 전환만 감지하는 구조여서 이벤트를 놓칠 수 있음을 확인.
- `ScratchState`에 `revealEventId`를 추가하고, `RevealParticle`이 해당 이벤트 ID 변화로 재생되도록 변경.
- 저장 버튼 활성화 기준인 `isCleared`는 그대로 유지하고, 시각 효과만 이벤트 기반으로 분리하여 재생 누락 가능성을 낮춤.

### 3. 광고 수익화 방향 재검토 및 정책 확정
- 전면광고, 배너광고, 보상형 광고의 차이와 앱 흐름 적합도를 검토.
- `anime_title_academy`는 반복 생성 중심 구조라 전면광고와 보상형 광고의 타이밍이 사실상 겹친다는 점을 확인.
- 이에 따라 메인 수익화 방식은 `보상형 광고`, 전면광고는 현재 구조에서 제외하는 방향으로 정리.
- 모델별 무료 이용권 및 광고 충전 정책 확정:
  - `빠름`: 하루 5회 무료 / 광고 1회당 3회 충전
  - `밸런스`: 하루 3회 무료 / 광고 1회당 2회 충전
  - `고품질`: 하루 1회 무료 / 광고 1회당 2회 충전
- 무료 이용권과 광고 충전 잔여량은 모두 매일 `00:00`에 초기화하도록 결정.
- 광고 실패 시 충전은 지급하지 않고, 광고 성공 시에는 현재 요청한 생성 흐름을 자동으로 이어서 실행하도록 정책 결정.

## 🛠 트러블슈팅
### 1. 모델 선택 기능 추가 시 LLM 경로 전파 범위 확대 필요
- **문제**: 결과 화면에서 모델 선택 UI만 추가해서는 실제 LLM 모델이 바뀌지 않았고, 선택값이 `AiClient`까지 전달되지 않는 구조였음.
- **해결**: `AiClient.generateText`에 모델 override 인자를 추가하고, datasource/repository/notifier/UI까지 선택 모델명을 전달하도록 파이프라인 전체 시그니처를 정리.

### 2. 스크래치 이펙트 재생 누락 원인 분석
- **문제**: 저장 버튼은 활성화됐는데 꽃가루 이펙트가 가끔 재생되지 않는 현상이 발생.
- **해결**: 저장 버튼과 이펙트 모두 `isCleared`에 묶여 있었지만, 이펙트는 `false -> true` 전환 이벤트에만 의존하고 있음을 확인. `revealEventId` 기반 이벤트 트리거로 변경하여 재생 누락 가능성을 낮춤.

### 3. 광고 방식 논의 중 전면광고와 보상형 광고 역할 충돌 확인
- **문제**: 반복 생성 시점에 전면광고와 보상형 광고를 동시에 고려하면, 같은 타이밍에 광고를 두 번 걸 수 있는 구조가 되어 UX가 나빠질 위험이 있었음.
- **해결**: 앱의 핵심 반복 행동이 `다시 생성`이라는 점을 기준으로 검토한 뒤, 메인 수익화는 보상형 광고만 유지하고 전면광고는 현재 구조에서 제외하는 방향으로 정리.

### 4. “Google 로그인으로 로그인한 계정의 Gemini를 그대로 사용” 가능 여부 검토
- **문제**: `flutter_base`의 Google 로그인/OAuth 구조를 이용해 사용자가 자기 계정의 Gemini 비용을 부담하게 할 수 있는지 검토 필요.
- **해결**: `flutter_base`의 GoogleAuthService / GeminiApiService 구조를 분석하고, 사용자 로그인과 Gemini API 과금 주체가 동일하지 않을 수 있음을 정리. 현실적인 방향은 `사용자별 API key(BYOK)` 또는 별도 Cloud project 기반 구조라는 결론을 문서화.
