## 🔬 버그 심층 분석 (Claude Code 수행)

### 문제 1: `flutter` 명령이 git bash에서 즉시 실패

- **원인**: puro가 설치한 `flutter` bash 스크립트(`~/.puro/envs/stable/flutter/bin/flutter`)는 내부에서 `$PURO_BIN/puro`를 호출하는데, puro의 실행파일은 `puro.bat`만 존재하고 bash에서 찾을 수 있는 `puro`(확장자 없음)는 WinGet 설치 경로에만 있음. 따라서 flutter bash 스크립트가 puro를 못 찾아 즉시 종료됨.
- **해결**: `flutter` 대신 `puro flutter`로 직접 호출. `puro` 바이너리는 bash PATH에 존재함(`which puro` 확인).

### 문제 2: `flutter run -d chrome`에서 `SDK root directory not found` 에러

이것이 핵심 버그. Dart 코드로 직접 재현해 원인을 특정함.

**버그 발생 경로 (Debug 모드 한정)**:

```
flutter run -d chrome (debug mode)
  ↓
resident_runner.dart:125
  sdkRoot = getHostArtifact(flutterWebSdk).path
           = "C:\...\flutter_web_sdk"   ← 정상 Windows 경로
  ↓
ResidentCompiler 생성 → frontend_server 프로세스 시작
  --sdk-root "C:\...\flutter_web_sdk/"
  ↓
frontend_server의 _ensureFolderPath("C:\...\flutter_web_sdk/")
  uri = Uri.file("C:\...", windows: true)  → file:///C:/...
  return uri.path   ← ❌ 버그: .path는 "/C:/..." (URI 경로 성분)를 반환
                       (올바른 방법은 uri.toFilePath() → "C:\...")
  ↓
Directory("/C:/Users/reals/.puro/...").existsSync()
  → Windows에서 "/C:/"는 유효하지 않은 경로 (OS Error 123)
  ↓
"SDK root directory not found: /C:/Users/reals/..."
```

**핵심**: `Uri.file(winPath).path` vs `Uri.file(winPath).toFilePath()` 차이

```dart
final uri = Uri.file(r'C:\Users\reals\.puro\...\flutter_web_sdk', windows: true);
// uri.path       = "/C:/Users/reals/.puro/.../flutter_web_sdk"  ← 잘못됨
// uri.toFilePath() = "C:\Users\reals\.puro\...\flutter_web_sdk"  ← 올바름
```

**버그 위치**: `dart-sdk/bin/snapshots/frontend_server_aot.dart.snapshot` 내부 `_ensureFolderPath` 함수. AOT 컴파일된 바이너리라 직접 수정 불가.

**왜 `flutter build web`은 되는가?** Build 명령은 `dart2wasm`을 직접 호출하며 frontend_server를 거치지 않기 때문.

**왜 `--profile`은 되는가?**

```dart
// resident_web_runner.dart:332
if (buildInfo.isDebug && !webUseWasm) {
    // DDC 인크리멘탈 컴파일러 사용 ← frontend_server 경유 → 버그 발생
} else {
    // WebBuilder.buildWeb() 사용  ← flutter build web과 동일 경로 → 정상
}
```

`--profile` 또는 `--release` 모드는 `else` 분기를 타므로 문제 없음.

## 🛡️ 재발 방지 및 주의사항

### 1. 이 환경의 근본적 제약

| 항목 | 내용 |
|------|------|
| 프로젝트 드라이브 | `E:` |
| Flutter SDK 드라이브 | `C:` (puro 관리) |
| 문제 발생 조건 | `flutter run -d chrome` (debug 모드) |
| 영향받는 컴포넌트 | `frontend_server_aot.dart.snapshot` (Dart SDK 내장) |
| 업스트림 버그 여부 | **Yes** – Flutter/Dart 툴체인의 Windows 크로스 드라이브 경로 처리 버그 |

### 2. 현재 개발 워크플로 (제약 내 최적)

```bash
# 앱 실행 (Chrome에서 확인)
npm run dev:ladder          # --profile 모드, 첫 빌드 후 Chrome 자동 열림

# 소스 변경 후 반영
# 터미널에서 r 키 → Hot Restart (전체 재빌드, 상태 초기화)
# 터미널에서 q 키 → 종료
```

> **Hot Reload 없음**: `--profile` 모드는 Hot Restart만 지원. 변경 후 `r` 키로 재시작.

### 3. Flutter SDK를 같은 드라이브에 두면 해결되는가?

**아니오.** 이 버그는 드라이브가 달라서가 아니라, `frontend_server_aot.dart.snapshot`의 `_ensureFolderPath` 함수가 Windows에서 `uri.path` (잘못됨)를 사용하기 때문임. 드라이브가 같아도 `/C:/path` 문제는 동일하게 발생함.

### 4. 이 버그가 수정되면? (향후 Flutter/Dart 업데이트 시)

`dart-sdk` 업데이트로 `frontend_server_aot.dart.snapshot`이 패치되면 `--profile` 없이도 debug 모드가 동작할 수 있음. 그 때 `package.json`의 `--profile` 플래그를 제거하면 됨.

### 5. `flutter` 대신 항상 `puro flutter` 사용

이 프로젝트에서 npm 스크립트로 flutter를 실행할 때는 반드시 `puro flutter`를 사용할 것. `flutter` 단독 명령은 bash에서 동작하지 않음(문제 1 참조).
