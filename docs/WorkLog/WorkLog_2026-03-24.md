## 실행 인자 / 설정 정리

### 1. 가장 자주 쓰는 실행 명령
#### PC에서 가짜 광고 흐름만 확인
```powershell
flutter run -d windows --dart-define=FORCE_QUOTA_AND_ADS=true --dart-define=REWARDED_AD_MODE=fake
```

이 명령의 의미:
- `FORCE_QUOTA_AND_ADS=true`
  - 디버그 기본 우회를 끄고, quota와 광고 흐름을 강제로 켠다.
- `REWARDED_AD_MODE=fake`
  - 실제 광고 SDK는 쓰지 않고, 광고를 봤다고 가정한 성공 시뮬레이션으로 흐름만 검증한다.

확인 가능한 것:
- 무료 횟수 차감
- 무료 소진 후 광고 유도 다이얼로그
- 광고 성공 후 충전
- 자동 재개

확인 불가능한 것:
- 실제 광고 화면 표시

#### Android에서 실제 테스트 광고 보기
```powershell
flutter run -d android --dart-define=FORCE_QUOTA_AND_ADS=true --dart-define=REWARDED_AD_MODE=test
```

이 명령의 의미:
- `FORCE_QUOTA_AND_ADS=true`
  - 디버그 기본 우회를 끄고 광고/제한 정책을 실제처럼 적용
- `REWARDED_AD_MODE=test`
  - 실제 광고 SDK를 사용하되, 실광고가 아니라 Google의 테스트 광고 ID를 사용

확인 가능한 것:
- 실제 테스트 보상형 광고 표시
- 광고 완료 시 quota 충전
- 자동 재개
- 닫기/실패 시 충전 없음

### 2. 각 인자의 실제 의미
#### `FORCE_QUOTA_AND_ADS`
- 기본 디버그 모드에서는 광고와 quota 제한을 우회한다.
- 이 값을 `true`로 주면 “디버그인데도 실제 정책을 강제로 켜라”는 뜻이 된다.

#### `REWARDED_AD_MODE`
- 광고를 어떤 방식으로 처리할지 정하는 값
- 지원 값:
  - `disabled`
  - `fake`
  - `test`
  - `production`

각 값의 의미:
- `disabled`
  - 광고 기능을 끔
- `fake`
  - 가짜 성공 처리
- `test`
  - 실제 SDK + 테스트 광고 ID
- `production`
  - 실제 SDK + 실광고 ID

## 추후 반드시 알아야 하는 개념

### 1. `App ID`와 `Rewarded Ad Unit ID`는 다른 값이다
- `App ID`
  - AdMob에 등록된 앱 자체의 식별자
  - Android Manifest에 들어간다
- `Rewarded Ad Unit ID`
  - 그 앱 안에서 만든 보상형 광고 슬롯의 식별자
  - Dart 쪽 광고 로드 시 사용한다

즉:
- `App ID`는 “이 앱이 누구냐”
- `Rewarded Ad Unit ID`는 “이 앱 안의 어떤 광고 슬롯이냐”

### 2. 왜 release에서 `App ID`가 없으면 빌드를 막아놨는가
- 테스트용 App ID가 릴리스에 그대로 들어가면 운영 사고다.
- 빌드 단계에서 바로 막아야 조용히 잘못 배포되는 일을 줄일 수 있다.
- 그래서 Android release task를 요청했을 때 `admob.android.app.id` 또는 `-PADMOB_ANDROID_APP_ID`가 없으면 빌드를 실패시키도록 설정했다.

이건 추후 반드시 알아야 하는 부분이다.
- 지금 당장 개발 흐름에는 직접 안 닿을 수 있음
- 하지만 실제 배포 직전에는 반드시 이해하고 있어야 함

### 3. 왜 `production` 모드에서 rewarded unit id가 없으면 앱 시작 시 바로 실패시키는가
- 실광고 모드인데 광고 단위 ID가 없으면, 사용자는 앱을 실행한 뒤 한참 지나서야 광고가 안 뜬다는 사실을 보게 된다.
- 그건 발견이 늦고, 운영 리스크가 크다.
- 그래서 `production`인데 rewarded ad unit id가 비어 있으면 앱 시작 단계에서 바로 실패시키는 쪽이 더 안전하다.

이것도 추후 알아야 하는 부분이다.
- 테스트 단계에서는 몰라도 됨
- 릴리스 전환 단계에서는 반드시 알아야 함

### 4. `AndroidManifest.xml`의 placeholder가 무슨 뜻인가
- 예전에는 Manifest 안에 App ID 문자열을 직접 써넣는 방식이었다.
- 지금은 `${admobApplicationId}`라는 자리표시자만 두고, 실제 값은 빌드 타입에 따라 Gradle이 주입한다.

이 구조의 장점:
- debug/profile은 테스트 App ID 자동 사용
- release는 실제 App ID를 강제
- 같은 파일에 테스트/실운영 값을 하드코딩하지 않아도 됨

즉 “placeholder”는
- 실제 값을 나중에 빌드 과정에서 넣기 위한 빈 칸이라고 보면 된다.
